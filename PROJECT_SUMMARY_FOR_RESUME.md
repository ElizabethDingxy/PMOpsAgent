# PMOpsAgent 项目总结与简历材料

本文档用于帮助你理解 PMOpsAgent 的完整工作流程，并把项目经历整理成简历、面试表达和作品集说明。

## 1. 项目一句话

PMOpsAgent 是一个面向中国团队的 AI 产品经理 Agent。它可以从 CSV 或飞书多维表格读取用户反馈，自动完成需求聚类、MVP 范围判断、RICE 优先级评估、PRD 草稿生成、研发任务拆解，并通过飞书和 TAPD 接入真实团队协作流程。

更直白地说：它不是一个普通聊天机器人，而是一个能把“散乱用户反馈”转化为“可评审 PRD 和可执行研发任务”的产品经理工作流助手。

## 2. 它要解决什么问题

产品经理日常会面对大量来自访谈、客服、问卷、社群、销售和运营的用户反馈。传统流程通常是：

```text
收集反馈
↓
人工阅读和归类
↓
判断哪些需求值得做
↓
写 PRD
↓
拆任务
↓
和团队评审
↓
创建研发任务
```

这个过程费时，而且容易出现几个问题：

- 反馈很多，但需求主题不清晰。
- PRD 依赖人工经验，证据链容易缺失。
- 优先级经常变成拍脑袋。
- 飞书文档、TAPD 任务、群内评审之间需要来回切换。
- 历史分析结果难以复用，新的项目和旧项目容易重复。

PMOpsAgent 的目标是把这套流程变成一个可追踪、可审批、可接入工具的 Agent 工作流。

## 3. 当前已实现的核心能力

### 3.1 用户反馈读取

支持两种反馈来源：

- 本地 CSV 上传或示例 CSV。
- 飞书多维表格读取。

CSV 会被解析成统一结构：

```text
id
user_type
source
content
created_at
```

这样后续 Agent 分析时，不关心数据来自 CSV 还是飞书表格，只处理统一的反馈数据。

### 3.2 多 Agent 分工分析

正式分析不是一次大 prompt 直接生成所有内容，而是拆成多个内部角色：

```text
Research Agent
↓
Strategy Agent
↓
PRD Agent
↓
Delivery Agent
↓
Orchestrator 汇总
```

各 Agent 的职责：

- Research Agent：读取用户反馈，提炼需求主题、用户痛点、证据反馈。
- Strategy Agent：判断 MVP 范围、RICE 优先级、风险和开放问题。
- PRD Agent：生成 PRD 草稿，包括背景、目标用户、问题陈述、目标、非目标、功能需求和成功指标。
- Delivery Agent：把产品需求拆成研发任务，并生成适合发到飞书群的评审摘要。
- Orchestrator：统一校验结构化输出、生成 trace、保存历史运行。

### 3.3 结构化输出

Agent 输出不是一整段 Markdown，而是结构化 JSON，包含：

- 产品名称和摘要。
- 需求聚类。
- MVP 范围。
- RICE 优先级。
- PRD 草稿。
- 研发任务草稿。
- 飞书评审摘要。
- 风险项。
- 开放问题。
- trace 执行过程。

这样前端可以稳定展示，也方便后续写入飞书文档、TAPD 或其它系统。

### 3.4 可解释性

项目增强了“为什么这么判断”的能力：

- 每个需求主题必须引用用户反馈 ID。
- MVP 范围要说明 must-have、should-have 和暂不做。
- RICE 优先级展示 Reach、Impact、Confidence、Effort 和计算结果。
- 低置信度判断会暴露出来。
- 研发任务可以追溯到用户证据和优先级理由。

这让 Agent 更像产品经理，而不是只会生成漂亮文本的 PRD 工具。

### 3.5 数据和指标补充

除了用户反馈，项目还支持补充业务上下文和指标 CSV，例如：

- 当前业务目标。
- 北极星指标。
- 成功指标。
- 埋点方案。

这些信息会进入多 Agent prompt，让 PRD 不只基于用户声音，也能结合业务目标。

### 3.6 Web Demo 工作台

网页端提供完整演示闭环：

```text
上传或加载反馈
↓
开始分析
↓
查看 trace
↓
查看需求洞察
↓
查看 MVP 和 RICE
↓
查看或编辑 PRD
↓
查看或编辑研发任务
↓
确认审批
↓
创建飞书 PRD / 创建 TAPD / 发送飞书摘要
```

网页上支持：

- 历史运行回放。
- PRD 草稿编辑。
- 研发任务编辑。
- 飞书摘要编辑。
- 审批状态管理。
- 创建飞书文档。
- 创建 TAPD 需求和任务。
- 发送飞书群评审摘要。

### 3.7 Trace 与历史回放

每次 Agent 运行都会生成 trace，记录每一步：

```text
读取反馈
分析需求
生成 MVP
生成 PRD
拆解任务
等待审批
外部工具调用
```

历史运行保存在：

```text
data/runs/*.json
```

前端可以加载历史结果，用于复盘、继续审批或查看详细分析。

### 3.8 审批与安全边界

项目明确区分“分析”和“写入外部系统”。

Agent 可以自动分析，但不能自动执行高影响写操作。以下动作都需要用户确认：

- 发送飞书群评审摘要。
- 创建飞书 PRD 文档。
- 创建 TAPD 需求和任务。
- 合并项目记忆。
- 通过或驳回分析结果。

这体现了 Agent 产品中的关键设计：AI 可以建议和准备动作，但真实写入要有人工确认。

### 3.9 飞书接入

项目接入了多种飞书能力：

- 飞书自定义机器人 webhook：从网页发送评审摘要到群。
- 飞书自建应用机器人：支持群里 @PMOpsAgent。
- 飞书事件订阅：接收群聊消息。
- 飞书消息回复 API：机器人在群里回复。
- 飞书多维表格 API：读取用户反馈。
- 飞书 OAuth：用授权用户身份搜索其可访问的 Base。
- 飞书新版文档 API：创建 PRD 文档。
- 飞书交互式卡片：在群聊里完成审批操作。

### 3.10 TAPD 接入

项目支持把 Agent 生成的 PRD 和研发任务同步到 TAPD：

- 查询 TAPD 公司下有哪些项目。
- 用户选择目标项目。
- 创建 TAPD 需求。
- 创建关联研发任务。
- 返回 TAPD 链接。

这让输出不只是“文档”，而是能进入研发协作系统。

### 3.11 飞书群聊自然语言 Agent

群聊中可以 @PMOpsAgent，用自然语言表达需求，例如：

```text
@PMOpsAgent 当前有哪些 Base？
@PMOpsAgent 帮我分析 demo-saas-feedback
@PMOpsAgent 自动销售记录摘要为什么是 P1？
@PMOpsAgent 当前记住了哪些项目？
@PMOpsAgent 合并 AI智能学情分析系统、智学助手、AI智能学习助手
```

当前架构已经从“固定关键词路由”升级为：

```text
用户自然语言
↓
轻量安全入口
↓
conversationAgent 理解意图
↓
LLM 规划工具链
↓
执行白名单工具
↓
需要写入时生成待确认动作
↓
用户确认后执行
```

轻量入口只做安全守门：

- 忽略机器人自己发的消息。
- 处理空消息。
- 标记高风险操作。

真正的意图理解和工具选择交给 conversationAgent。

### 3.12 产品上下文记忆

PMOpsAgent 会把每次分析沉淀成“产品项目记忆”，保存在：

```text
data/product-memory/*.json
```

记忆包含：

- 项目名称和别名。
- 来源 CSV / Base / 表格。
- 最近运行 ID。
- PRD 摘要。
- MVP 范围。
- RICE 判断。
- 研发任务。
- 用户证据。

这样用户后续可以继续追问：

```text
@PMOpsAgent 销售助手项目里，自动摘要为什么是 P1？
@PMOpsAgent 针对 demo-saas-feedback，重新评估优先级
```

Agent 不需要重新从零开始，而是先读取项目记忆，再结合新问题回答。

### 3.13 项目记忆合并

项目支持处理重复项目记忆。比如同一个产品可能来自 CSV，也可能来自飞书 Base，名称或来源不同，但本质是同一个项目。

当前合并流程不是简单查重，而是统一工具链：

```text
read_product_memories
↓
compare_product_memories
↓
prepare_merge_product_memories
↓
用户确认
↓
execute_pending_action
```

也就是说，用户说“合并这两个项目”时，Agent 会先读取最近对话和项目记忆，判断这些项目是否本质重复，说明合并依据和风险，然后生成待确认动作。只有用户确认后才真正写入合并结果。

## 4. 端到端工作流程

### 4.1 网页端工作流

```text
用户上传 CSV 或读取飞书表格
↓
前端解析或服务端读取反馈
↓
/api/analyze 调用 runAgent
↓
多 Agent 串行分析
↓
结构化结果校验
↓
保存历史运行
↓
更新产品项目记忆
↓
前端展示 trace、洞察、PRD、任务
↓
用户编辑并确认
↓
用户选择创建飞书 PRD、创建 TAPD 或发送飞书摘要
```

### 4.2 飞书群聊分析工作流

```text
用户在飞书群 @PMOpsAgent
↓
飞书事件订阅请求 /api/feishu/events
↓
eventBot 校验消息并清理 @ 文本
↓
conversationAgent 理解意图并规划工具
↓
读取飞书 Base / 表格 / 示例反馈 / 项目记忆
↓
调用 runAgent 生成分析
↓
保存历史运行和项目记忆
↓
回复群聊摘要和审批卡片
```

### 4.3 飞书内审批工作流

```text
Agent 分析完成
↓
发送飞书审批卡片
↓
用户点击通过 / 驳回 / 创建 PRD / 创建 TAPD
↓
/api/feishu/card-actions 接收卡片回调
↓
统一 action executor 执行对应动作
↓
更新审批状态和卡片
```

### 4.4 群聊自然语言确认工作流

```text
用户说“帮我创建 PRD”或“合并这几个项目”
↓
conversationAgent 判断这是高影响操作
↓
准备 pendingAction
↓
回复用户确认内容
↓
用户回复“确认”
↓
execute_pending_action 执行真实写入
```

## 6. 为什么它更像 Agent，而不是普通 AI 聊天框

普通聊天框通常是：

```text
用户问
↓
模型答
```

PMOpsAgent 的核心区别是：

- 有明确任务目标：从反馈到 PRD、任务和评审。
- 有结构化输出：不是纯文本，而是可展示、可校验、可写入系统的数据。
- 有工作流：读取、分析、生成、审批、写入。
- 有 trace：用户能看到每一步做了什么。
- 有工具调用：飞书、TAPD、多维表格、文档、卡片。
- 有安全边界：高影响操作必须确认。
- 有记忆：可以基于项目上下文继续讨论，而不是每次重来。
- 有多 Agent 分工：不同阶段由不同角色处理。

## 9. 面试时可以这样讲

### 9.1 项目背景

我做的是一个 AI 产品经理 Agent，目标是解决产品经理从用户反馈中提炼需求、判断优先级、写 PRD 和拆研发任务的重复劳动问题。

### 9.2 核心流程

用户可以上传 CSV，也可以让机器人读取飞书多维表格。Agent 会先做需求聚类，再判断 MVP 范围和 RICE 优先级，然后生成 PRD 和研发任务。结果会保存在历史运行里，并可以通过飞书卡片或网页完成审批，审批后再创建飞书 PRD 或 TAPD 任务。

### 9.3 为什么是 Agent

这个项目不是简单调用大模型生成 PRD。我做了结构化输出、trace、工具调用、审批、人类确认和上下文记忆。它能规划工具链，例如读取项目记忆、比较项目是否重复、准备合并动作，再等用户确认后执行。

### 9.4 技术难点

主要难点有三个：

1. 如何让 LLM 输出稳定可用的结构化结果。
2. 如何把飞书、TAPD 这些真实工具接入到安全的审批流里。
3. 如何让群聊机器人从固定命令升级为能理解上下文并组织工具链的 Agent。

### 9.5 你的产品思考

我在设计时没有让 Agent 自动写入外部系统，而是把高影响操作都设计成 pending action。也就是说，AI 可以分析和准备动作，但必须由用户确认后才真正创建文档、创建任务或合并记忆。这是为了让 Agent 在真实团队协作里更可信。

## 10. 后续可以继续完善的方向

如果继续做成更完整的产品，可以补这些能力：

- 数据库存储：替代本地 JSON，支持多用户和线上部署。
- 用户与项目权限：区分不同团队、项目和成员权限。
- 更完整的产品上下文：保存竞品、路线图、版本目标、历史决策。
- PRD diff：让用户在飞书群里讨论后，Agent 生成可审阅的修改差异。
- TAPD 双向同步：读取已有 TAPD 需求状态，避免重复创建。
- 指标回流：接入埋点或 BI 数据，评估上线后是否达成目标。
- 评审纪要：把飞书群讨论沉淀成 PRD 决策记录。
- 部署到云服务器：配置 HTTPS、Nginx、PM2、持久化 data 目录。

## 11. 当前项目边界

当前版本仍然是面试作品 Demo，不是完整 SaaS 产品。它已经能展示 Agent 核心能力，但还有这些边界：

- 使用本地 JSON 存储，不适合多人长期生产使用。
- 没有登录系统和团队权限模型。
- 依赖飞书、TAPD 和 DeepSeek 的外部 API 可用性。
- 部署时需要有持久磁盘保存历史、OAuth token、项目记忆和会话状态。
- 对复杂 PRD 修改还可以继续增强 diff、版本管理和审批记录。

## 12. 一句话总结

PMOpsAgent 展示的是一个从“AI 生成内容”走向“AI 参与产品工作流”的 Agent：它能读取反馈、做产品判断、生成交付物、调用协作工具、保留上下文记忆，并在关键写入动作前请求人类确认。
